### 计网整理

> 这一份笔记仅供课后整理思路用，只记录一些比较重要的东西来理顺知识框架，有些细节还是在slides上。在复习的时候，可以以slides为主，配合这一份笔记食用。也会记录一些考点之类的。

#### Ch1 Introduction

- 网络边缘host+router&switch+links 构成network 而Internet是局部的network互联形成的：网络的网络
- protocol 协议：定义了二进制数据流的传输格式 （二进制的内容如何解读），是共同遵守的约定
- network edge:  host **包括client和server！** 用户在使用的end systems 都是网络边缘
- access network **接入网络连接网络边缘和网络核心！** 理解为有线/无线的通信链路。
- 连接end system 与 edge router(边缘路由器) ：有residential接入网络 机构接入网络 移动接入网络（WiFi和5G等） 
- cable-based 电缆型接入网络：电缆接入每个终端，最后**汇总到cable headend总线**，最终到ISP的主干。使用了FDM 频分复用，将频率分成子频段，每个频段信号不同，设备之间share cable headend **共享带宽！**。使用HFC 混合光纤同轴电缆 具有asymmetric性质，downstream会更多。
- DSL 电话线接入，是专用的线路接入，不再分频复用，**home之间不共享**。
- wireless access network 无线接入，**共享基站**。WLANs 由edge router接入Internet，范围较小；cellular由大型基站进行广播，范围较大 4G/5G
- 企业接入网络：设备较多，先连接到交换机，形成局部的子网(**ethernet**) 然后再汇总到路由器，最终到ISP
- 终端发送**packet形式的data** 每个packet的长度是$L$ bits，**把数据推上链路的**transmission rate（**传输速率，也就是bandwidth**）是$R$ bits/sec 那么最终的传输时延 **transmission delay=$\frac LR$** 
- 注意transmission和propagation delay的区别。transmission指的是**把数据从接入网络推上链路**的时间，可以看成第一个bit和最后一个bit上link的时间差，和主机之间距离无关。propagation则是在主机之间传播的时间。**这两者要会区分 会计算！** 同时，注意kbps转成bps是乘1000的，不是1024 单位要注意！
- Link的物理媒介：传输网络信号，分为有线guided和无线unguided。同轴电缆 光纤是有线的，光纤fiber optic 的好处：①带宽高，传输速度快 ②错误率低，免疫电磁波的噪声； 也有无线广播式的，如WLAN wide-area cellular 等
- 网络核心：**路由器交错形成的网状结构** 主干网络就是路由器形成的集合 路由器有edge router和core router
- 网络核心的2个功能：①forwarding转发，从路由器的input link接收数据，查表得到应该去到哪一个output link  ②routing 路由：使用特定的算法计算路由表格，判断应该向哪一个输出端口（子网）发送数据
- packet-switching**分组交换**：将数据拆分成多个packet，在网络核心的路由器中**逐跳转发**，在路由器之间的链路上以最大带宽R发送，最终到达destination 这是一个**store-and-forward** 的过程，先存储再转发。分组交换的以下三点和对应的slides还是很重要的，复习时候重点看看。
- 在分组交换的过程中，**从路由器发到链路上的时候会存在transmission delay**，因为link上是有带宽限制的，不能一次性全部转发，因此会有L/R的一个delay 其中R是link的带宽
- 注意存储-转发机制的一些细节。数据来到路由器后，首先存储在buffer中，由路由器判断往哪个出口发送。因此（当arrival rate 大于 transmission rate的时候）会有一个排队等待的机制，轮到了才能发送。同时，buffer的容量有限，如果超出最大容量则发生**packets lost** 丢包了，需要重传。
- 与之相对的circuit switching 电路交换，这个就比较简单了，直接end-end 传输数据了，但是会造成很多线路处于idle的状态，浪费资源。因此有FDM和TDM两种方式，拆分频率与时间。
- packet switching 分组交换使得多用户共享，且有排队机制，因此利用率比电路交换更高。
- 互联网结构：局部地区的host形成小型的子网，也就是**access network** 再进一步通过ISP接入互联网。多个ISP之间也要互联（使得任意两台host都可以通信） ISP之间通过IXP Internet exchange point 进行互联，同时也会出现地区性的regional ISP优化。对于一些应用，也会打造专门的服务器Content Provider network，作为专用设备减少对服务器的依赖。最终形成P46的互联网图。
- layering 分层管理，模块化，层之间只需要通过接口进行交互。上层依赖于下层提供的service。
- Internet protocol stack **按照协议进行分层** **application：**支持网络的应用 **transport**：进程与进程之间的数据传输，包括sender的break into packets和receiver的reassemble  **network：**将数据经过路由转发，从source到destination **link：**相邻节点之间的data transfer（路由器之间，主机与路由器之间）**physical：** 在物理的medium上传输bits 每一个的作用尽量记住，表述严谨。
- slides上举了一个例子，复习的时候可以跟着过一遍看看细节。学习的过程中，分层的思想也很重要。先对每一层有一个big picture 进行大致的了解，然后在学习具体的某一层的时候，思考上层需要这一层提供什么服务，下层为这一层提供了什么服务。而提供的服务是基于protocol的。
- delay相关的概念，比较细节，可能会出计算题！四种概念的时延要清楚，**同时要知道是在什么位置上发生的，心里要有那一张图！**
- ① queueing delay 在router的buffer中排队的时候会有排队时延，发生在arrival rate大于 link capacity(带宽)的时候 如果超出buffer还会发生packet loss 
- ② processing delay 处理时延，就是在路由器中看IP地址的首部，决定发往哪个出口的处理时间
- ③ transmission delay 路由器把一个packet的数据发送到link上的时间，transmission rate 就是link的带宽 link capacity
- ④ propagation delay  路由器在链路上传播的时间。可以类比高速收费站和路上，这一段是在路上的时间。
- throughput 吞吐量：单位时间从sender到receiver的数据量 有instantaneous和average的，一定要注意单位是Bps还是bps，考试就喜欢考这些的。 吞吐量有bottleneck link，多段链路中会受**最小吞吐量的限制**
- 后面讲了一些网络安全和历史，复习的时候就不看了。可以稍微记一下，security这一块 sniffing劫持信息 IP spoofing伪造身份  DoS拒绝服务攻击

#### Ch2 Application Layer

- 创建一个网络应用：在需要网络通信的时候调用transport layer 提供的**API**就可以，不需要写具体如何使用网络核心。
- 应用层的paradigm1： 客户端-服务器模式。有一个长期运行，**固定IP地址**的服务器，提供数据。其余客户端向server索取服务，IP地址可以是不确定的，**而且客户端之间的通信也必须经过服务器，没有直接端到端的** HTTP和IMAP协议都是这种模式的。
- 应用层的paradigm2： P2P模式。是较早期的，不存在一个固定的服务器，每个host同时承担server和client的角色，节点端到端直接通信。
- 本质上，不是主机之间在通信，而是**processes communicating** 也就是进程之间的通信。每个应用都是跑在进程上的。在同一台主机上不同进程的通信，用内置的进程通信方法，由操作系统定义；不同主机通信才需要network来通信，exchange message
- 进程的identifier **由IP+port-numbers构成！** IP定位主机，port定位主机上的**具体进程！**比如，HTTP服务器的端口号是80，mail服务器是25
- 应用层协议，定义了 message种类，message语法（有什么字段，如何排列），message语义，进程send和respond的规则。
- 传输层协议：包括TCP/UDP为应用层提供服务，**核心作用是通过IP+port提供进程到进程的通信！**TCP协议是reliable transport 在进程之间更可靠，而且有flow control, congestion control之类的保护机制，但是也有代价：是connection-oriented的，需要提前建立连接，速度较慢。UDP不可靠，会丢包，但是不需要提前建立连接(connection set-up)，速度会快一些。 在不需要完全可靠，追求速度的场景，比如直播和游戏用UDP
- Web page：一系列的object（文件）在浏览器上进行渲染。base HTML文件成为网页的基础，引用其他资源，使用URL进行定位。URL是host name+path name（文件路径）的形式。
- HTTP协议在client端：生成request 然后server端解析request并生成response 再回到client端解析response这个就是核心流程。在client和server之间的网络传输，底层是**基于TCP协议的！**：首先客户端生成TCP connection发送，服务器接收，实现了message exchange之后TCP connection关闭。注意，HTTP协议是**stateless**的，也就是两条HTTP request之间是独立的，不包含彼此的信息。
- HTTP connection的两种类别：**Non-persistent HTTP1.0** TCP连接打开之后 **最多只能在这个TCP connection上发送一个object**，然后就会关闭了。 **Persistent HTTP1.1** TCP连接打开之后可以在这上面发送多个object，才会关闭。keep-alive 机制，在代码中显式声明/超时/出现异常等才会关闭这个connection！这种复用TCP connection的机制，减少了**往返时间RTT**
- RTT：round-trip time 也就是一个packet从client出发到server再回来所用的时间，HTTP1.0**对于每个object而言，都需要两次的packet往返，也就是2RTT！** 但是HTTp1.1 复用了之前的TCP connection，不再需要建立连接的RTT了，效率提升了。
- **HTTP request message**：请求类http的详细信息。需要理解的信息的含义。首先是一行**request line**，会说明这一条request是GET POST HEAD类型的，后面跟**路径path，指定要访问哪一个资源，不是完整的URL**，最后是HTTP version。 后面的内容都是**header line**（首部！）包含HOST **目标的域名！**ACCEPT是可以接受的文件格式，还有语言 压缩 编码的格式。KeepAlive是希望TCP connection一直保存的时间（对于HTTP1.1而言），最后空行表示结束了接下来是body
- 几种不同的request message：**POST**将一些信息输入到网页中，客户端将信息放在request的body中，上传给服务器；**GET** 得到数据，可以在后面带？获取具体的资源；**HEAD**类似于GET但是只返回headers；**PUT**是上传文件进行**覆盖**指定URL的内容，POST只是新建并上传，PUT是覆盖原有内容。
- **response message** 第一行是**status line**告知协议和**status code** 后续的header line包含Last-modified 上一次修改的时间（后面代理的知识：如果你是第二次访问，还会多一行If-Modified-Since，就是说**本地已经有缓存了！**如果发生了更新（200）就需要重新发数据，如果没有更新（304）就不发内容了，直接加载本地缓存就可以（作业题里写过）还有**content-length**也是一个很重要的指标：后面的body部分需要这个量**来判断是否结束**
- cookie机制：用于保持客户端或服务器在传输过程中的**一些状态**。首次访问的时候，respond会返回set-cookie:id=12345 为每一个用户生成独特id，**存储在服务器的数据库中，返回时也在浏览器的缓存里！**后续再次访问的时候，浏览器自动带上cookie的id再做GET，服务器比对数据库的值就知道你的身份了。
- 网络缓存web cache和代理服务器proxy server: 代理服务器作为中间人，是客户端的服务器，是origin服务器的客户端。客户端先向cache发送HTTP request，如果缓存里有需要的文件就命中了，直接返回；如果没有，**cache先向origin server请求数据，收到回复之后再return给客户端**。
- conditional GET：利用缓存机制进行优化。浏览器缓存中有Last-modified，此时向proxy server发送请求的时候会带上**If-Modified-Since**，查看是否修改。只有修改了才需要重新向origin请求数据。
- HTTP/2 ：之前的HTTP1.1协议是复用TCP连接，但是**所有的请求都in-order按照先到先服务的顺序传输**，就会出现HOL block**对头阻塞**的问题：大的文件先传递了，后续的小文件才可以传递，同时，丢包之后也必须等它重传了才可以继续后面的。HTTP2则可以多路复用，一个TCP连接上同时跑多个请求，并且可以自己指定顺序。但是，底层的**TCP仍然会丢包重传，还是会有HOL问题**，这时候基于UDP的HTTP3就来了，彻底绕开。
- Email的主要组件有user agent 和 mail server 前者是客户端，也就是阅读邮件的地方；后者是服务器，起到接受邮件和发送邮件的双重身份。**在mail server之间的协议是SMTP 用于传递邮件报文**
- 发送邮件的过程：从Alice的user agent到Alice的server，使用SMTP协议；**从Alice的server到Bob的server，也是使用SMTP协议，并且通过@后面的内容找到对方的server**；从Bob的server到Bob的UA，也就是**邮件的接收环节，使用IMAP协议**。特别注意mail access protocol is Internet Mail Access Protocol ！！如果使用网页浏览器的话，在UA发送给server以及server接收给UA的时候，都需要用HTTP！
- 详解SMTP协议，与邮件的发送和传输有关，底层**基于TCP协议**（更加稳定一些，需要建立TCP connection的时间！），端口号是25 传输的时候，直接从server传到目标的server，没有经过中间的什么东西。SMTP传输分为三个phases：handshaking建立连接，transfer message，最后closure
- SMTP是push型的协议，也就是邮件的发送方主动进行message的推送；HTTP是pull型的协议，也就是客户端主动从服务器拉去信息。这是最本质的区别。同时，SMTP会使用Persistent connection，**一次TCP连接可以发送多个邮件！**
- mail的message格式也是有规范的，包括header和body，中间用一个空行隔开。
- DNS 本质是一个**应用层的协议**，基于UDP，端口号是53。可以**查询**IP地址和域名在**分布式数据库中**的双向映射关系。提供域名解析的服务。 DNS的主要作用有：name IP translation; aliasing 别名找到真名的服务；邮件地址的解析（@后面对方的服务器，就是使用DNS来解析目标的）；负载均衡 防止访问压力过大
- DNS的背后是**a distributed and hierarchical database** 有顶级域名 二级域名 三级域名等。如何理解hierarchy？**DNS的name server是分布式的数据库，也提供了查询接口，分为Root DNS server根服务器, Top level domain DNS server顶级域服务器, Authoritative DNS server权威服务器** 整体呈现树状结构，访问时首先访问根服务器，可以获知在哪一个branch，然后去访问对应的顶级服务器。 
- 理解课件上的例子，不要对应错了！要访问www.amazon.com 首先**访问Root，获知.com在哪个顶级域服务器，然后访问顶级域，获知amazon.com在哪个权威服务器**
- Root name server：全球13个服务器以及一些复制的服务器，本质上是分布式的数据库，里面存放主要是NS记录（也就是用于指路的，指向.com这个域名由哪一个顶级服务器来管，就可以走对应的分支了！）（NS记录和A记录不同，NS主要是指路，A记录则是返回真正对应的IPv4地址）
- Top Level Domain server(TDL)：顶级域名服务器，包括.com .net .edu以及.cn等国家的，也是负责NS记录再往下层的找权威域名的。举例：所有.com都在这个TDL server了，然后存了amazon.com youtube.com这些NS记录，可以访问具体的权威域名。
- Authoritative server： 权威服务器，**真正存储域名到IP地址的映射关系，并返回最终的答案！**里面存放的是A记录：也就是www.example.com → 93.184.216.34这种的，还会存放一些MX记录（邮件）和CNAME记录（别名）或者AAAA记录（IPv6） 注意，不管前面还有多少级，都归属同一个权威服务器管理，也就是所有的子树都是属于同一个权威服务器的，直接返回答案。
- 本地DNS服务器与缓存：Local DNS name server本地的服务器。每个ISP（比如运营商，学校）都有一个自己的Local DNS，当客户端进行一次DNS的查询，query首先会向local cache中查询有没有**recent name-address pair** 如果过期了或者没有命中，再去Root从头开始查找。只要某次查到结果，local就会存在cache中，但是缓存中的记录也有TTL，只会存在一段时间。如果进行了IP地址的更新，需要等待所有缓存都out of date 才可以正常访问。
- 域名查询有两种方式 ① iterated 迭代式，每一步都是“指路”，首先host发送query给local服务器，然后local服务器查询root，**root只会返回下一步在哪里查询，下一步还是由local DNS向顶级域名服务器发起的查询**，最后再由local DNS把查询到的结果返回给host  ② recursive递归式，local发送query查询root之后，**不返回结果给local，而是直接向下递归树**，直到查到最终结果才返回。  实际场景下一般是**混合使用**：客户端到local服务器用递归式，减小客户端操作难度；服务器之间是迭代查询，减轻外面服务器的压力。
- DNS records类型：DNS的数据库里面存储的是**resource records（RR）** format是(name,value,type,ttl)   有4种types A类型的value是IP地址，NS类型的value是返回去哪里查（如权威服务器）CNAME类型**name是alias，表示别名，value是canonical，表示真名** MX类型的value表示邮件的server
- DNS protocol message的报文格式：有query和reply两种，但是format是一致的。看看slides上的图，理解各部分的含义：identification里面有ID，**匹配query和reply，两者要使用同一个ID**；flags里面有一些标识，query or reply？ RD是否递归 RA是否支持递归 AA是否来自权威服务器(不是的话可能来自cache) 后面是具体的query和answer的内容。
- RR的插入与胶水机制：创建了一个新的RR比如liyun.com,就需要插入到TLD server里面，不需要插入到Root里面，因为看到.com就知道要往管理.com的TLD里面发了，而真正需要做的是TLD里面到liyun.com的权威服务器。因此，插入RR的时候**会向TLD里面插入两条记录：NS与A** nw.com → dns1.nw.com（NS） 以及dns1.nw.com → IP（A, glue）第二条作为胶水记录！如果只有NS，会出现找dns1.nw.com的时候仍然需要访问nw.com 造成循环依赖，而这个glue A则直接提供了dns1的IP地址，使得可以直接向后查询，**相当于提供了捷径打破循环依赖！**
- P2P架构，没有总是在线的服务器，终端之间直接通信。**self scalability**可拓展性很高，但是数据传输不稳定。
- P2P和cs架构的对比传输时间，有一些计算公式需要掌握。服务器-客户端传N份文件，需要考虑上传端和下载端，上传端的时间是NF/u 因为有N个copy 下载端的时间是F/di 理解一下，取最值就是最终的文件传输时间。而P2P就是不一样的思维了，首先服务器必须上传一份，时间F/u 而且每个用户必须下载一份，时间瓶颈F/u_min 同时，**总上传的份数要求是NF**，这一部分的带宽由所有用户承担！可以理解为P2P是在向其他人要文件的，所以所有人**都在利用上传的带宽！** 也就是时间为$NF/(u_s+\Sigma{u_i})$  在这三个瓶颈中取最值就是P2P的文件传输时间。因此，P2P的可拓展性是更强的，随着用户数量N的增大，cs架构线性增长，但是P2P架构趋向于平缓。
- BitTorrent 应用层协议，规范P2P。文件被划分成为小块chunks，在同一个群组torrent中，客户端互相交换这些chunks，多一个人就多一分上传的带宽。其中，tracker记录有哪些peer在当前这个torrent里面，可以传输这个文件。但是有完整文件了，这个客户端可能会直接离开这个torrent，所以也不一定稳定。
- streaming video 数据量大，如何解决传输问题？在client-side,主要有三点：使用coding进行时间或者空间上的压缩；使用buffer进行缓存，在网速快的时候多缓存一些，网速慢的时候用buffer解决连续播放问题；使用DASH进行客户端自适应的决策，决定何时接受chunk使得buffer的存储大小适中，根据网速决定接收什么清晰度（码率）的视频。
- 在服务器端，视频流经常会面临高并发的场景！因此，**需要CDN 内容分布式网络**来解决。将视频内容拷贝多份，分别存放在不同位置的站点中（分布式存储）。enter deep和bring home 是CDN的两种实现方式，前者更加深入access network，延迟低且单个站点的规模小，后者规模更大成本低。
- 注意从支持CDN的服务器下载数据的流程：首先client获取到manifest文件，里面有CDN的IP地址和chunks的码率等信息，CDN根据网络速度和拥塞情况，自主选择合适的版本后下载。**由于CDN的分布式服务器里都是内容相同的拷贝，所以可以下载的内容是一样的！**并且中途如果遇到了网络拥塞的congestion情况，也可以中途动态切换CDN服务器。
- socket编程：socket是应用层与传输层之间的管道，通过socket调用网络的底层基础设施。底层的传输层有UDP和TCP两种。UDP是不用提前建立连接的，所以服务器只需要**bind对应的本地端口，然后这个端口的信息都交给这个socket接收了**。然后只需要一直recvfrom等待传输。客户端则**设置好对方IP+port后，将数据使用sendto传输到对应的服务器**就可以。服务器接收后再传回客户端。就是不停sendto和recvfrom这样。
- TCP要建立连接（connect）会更加复杂一点。首先server也是需要**bind对应的本地端口作为专门收发信息的**，然后需要开始监听listen()，等待TCP连接。在客户端那边，需要主动connect到对应的服务器IP+port **相当于是TCP建立connection的过程！**当服务器收到连接后accept()，会**保存一个代表连接的socket，也就是connectionSocket()**,这是一条**专门的通道，用于和这个特定的客户端进行通信！** 随后服务器就可以用connectionSocket()接收并且发送了，而客户端由于已经connect上去了，所以直接用原来的clientSocket进行recv和send就可以。 还要注意使用encode和decode进行字符串和字节之间的转换，在网络传输的只能是字节！

#### Ch3 Transport Layer

- 传输层主要是提供了不同主机之间 进程与进程之间的通信。process-to-process communication 传输层具体职能上分为sender和receiver，前者将message拆分成segment**传给网络层**，后者将segment重组成message并**传给应用层**。
- 传输层和网络层的区别。传输层是**进程之间的通信**，网络层则是**主机之间的通信**，一个主机包含有多个进程。
- 注意receiver的实现细节：从网络层获取segment并检验，去掉header成message，最后**通过绑定在对应端口号上的socket，将message传到对应的进程！** 明白process和port的关系。
- TCP协议：可靠、按顺序发送，有拥塞控制和流控制，与之相比的UDP则不可靠，无序，也没有相应控制。但是两者都没有对delay和bandwidth的保证！因为底下网络层的IP协议只提供best-effort的服务。
- 在sender处进行multiplexing：将多个socket（多个进程）的数据进行处理，加上传输层的header之后，统一交给网络层IP进行传输。
- 在receiver处进行demultiplexing：主机收到网络层IP发来的datagrams，每个包都会有sourceIP和destIP，并且包里还会含有一个**TCP/UDP的segment**，而segment的header又含有sourcePort和destPort，这样主机就可以使用**IP+port**来确定这个数据是要交给哪个socket了(socket和端口是绑定的，而socket又属于某个进程，所以相当于将数据传到了对应的进程，是进程之间的通信)（TCP需要四元组！）
- 在socket编程里，每个socket都要绑定bind对应的本地端口。如果是UDP接收，无需提前建立连接，**只要是dest port相同，就全部交给一个socket进行处理！** 因为是无连接的协议，不用知道是从哪里来的，全部放到一个里面就够了。而UDP类型socket在发送的时候，必须指定对方的IP和port才可以。
- TCP的demux过程与UDP不同，必须加以区分（考试肯定会考的！TCPdemux需要四元组，UDPdemux只需要dest port，并且原因要很清楚）TCP是基于connection的协议，所以一个TCP socket对应的是一个connection，而不是一个port。所以只有一个dest port是不够的，**这样就无法区分是谁的请求了！**可以好好看看课件p26的图片，很多连接都是以port B,80作为目标的，但是他们的来源不同，所以不能像UDP一样简单粗暴的放在同一个socket里面，必须分别放在不同的socket中，这样才符合TCP的特点！因此，**需要使用TCP四元组，包含source IP+port以及dest IP+port，这样才可以唯一确定一个应该前往的socket，也就是确定对应的进程了。**
